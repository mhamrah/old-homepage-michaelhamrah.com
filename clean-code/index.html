Error Kernel Pattern

Basically, it is just a simple guideline you should always try to follow, stating that if an actor carries important internal state, then it should delegate dangerous tasks to child actors, so as to prevent the state-carrying actor from crashing.

Sometimes new child actors, not a necessity.

Control Dependencies

Separate Concerns

How Easy is it to Replace X?

The Dependency Rule: Source code dependencies can only point inwards.

The name of something declared in an outer circle must not be mentioned by the code in the an inner circle. That includes, functions, classes. variables, or any other named software entity.

This is the outcome of the Rmq refactor

Lean Architecture

DCI: Data, Context, Interaction

Split what a system is (data) from what a system does (interaction)

Context is the glue

Think of a waiter in a restaurant

Scala Paradigms

Cake Pattern? Self Types? SubCut?

All of these are just easier ways of wiring up traits and classes

The real question is testability. How easy is it to instantiate X. To test X.

TESTING FIRST WILL TELL YOU THIS!

Conforming to these simple rules is not hard, and will save you a lot of headaches going forward. By separating the software into layers, and conforming to The Dependency Rule, you will create a system that is intrinsically testable, with all the benefits that implies. When any of the external parts of the system become obsolete, like the database, or the web framework, you can replace those obsolete elements with a minimum of fuss.
